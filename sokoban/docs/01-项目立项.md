## 一、项目背景

### 1.1 立项动机

推箱子（Sokoban）诞生于 1981 年，由日本游戏公司 Thinking Rabbit 的 Hiroyuki Imabayashi 设计，是世界上最经典的益智游戏之一。玩家扮演仓库工人，将所有箱子推到指定目标点，是"推箱子"类游戏的鼻祖。

选择它作为本项目库（Web_Game_01）的第五款产品，原因有三：

1. **技术代表性**：推箱子是网格 + 状态机 + 撤销栈 + 死锁检测的"集大成者"，几乎涵盖所有 2D 益智类游戏算法。
2. **教学价值**：XSB 格式是行业标准，撤销/重做 + 死锁检测是经典算法课题。
3. **复用性强**：项目骨架（React + Zustand + Canvas + Vitest）继承自 snake / 2048 / tetris / flappybird，是项目库"v2.0 质量优先栈"的第五款落地。

### 1.2 项目定位

- **类型**：2D 网页益智游戏
- **风格**：经典木地板/砖块质感（明亮色调）+ 暗黑紫外框（项目库设计语言）
- **目标平台**：桌面浏览器（Chrome / Edge / Firefox / Safari）+ 移动端浏览器（触屏滑动）

## 二、项目目标

### 2.1 学习目标（核心）

| 序号 | 能力 | 衡量标准 |
| --- | --- | --- |
| 1 | XSB 关卡格式解析 | 严格按 sokobano.de 标准实现 + 单元测试覆盖 |
| 2 | 推箱子核心算法 | 移动 / 推箱 / 撞墙 / 死锁检测 |
| 3 | 撤销 / 重做 | UndoStack 栈式管理 + 状态快照 |
| 4 | 死锁检测 | 角落死锁 + 边墙死锁（覆盖 90% 常见情况） |
| 5 | 多关卡管理 | 解锁 / 通关 / 最佳记录持久化 |
| 6 | Canvas 2D 渲染 | 7 种 CellType + 动效（绿光闪烁 / 抖动） |
| 7 | 状态机 | menu / playing / paused / over 四阶段 |
| 8 | 触屏控制 | 滑动 + ActionBar 双输入 |

### 2.2 产品目标

- ✅ 交付一款可在浏览器流畅运行的 Sokoban 游戏
- ✅ 实现 PRD-05 中 MVP 全部 10 项功能（F-01 至 F-10）
- ✅ 完成至少 5 项 V2 扩展（F-11 撤销 / F-12 死锁检测 / F-13 关卡选择 / F-14 暂停 / F-15 音效）
- ✅ 内置 20 关 Microban 风格关卡
- ✅ 可通过 Vercel 一键部署

### 2.3 质量目标

| 指标 | 目标 |
| --- | --- |
| 性能 | 60fps，无掉帧 |
| 兼容性 | Chrome 100+ / Edge 100+ / Firefox 100+ / Safari 15+ / 移动端主流浏览器 |
| 代码质量 | TypeScript 严格模式，ESLint 通过，核心算法单测全绿 |
| 体积 | 首屏加载 ≤ 500KB（gzip） |
| 测试覆盖 | 推箱 / 撤销 / 死锁 / 关卡管理 100% 覆盖 |

## 五、风险评估

| 风险 | 概率 | 影响 | 应对措施 |
| --- | --- | --- | --- |
| 推箱子算法 bug | 中 | 高 | 严格按 XSB 标准，单元测试逐 case 覆盖 |
| 撤销栈内存膨胀 | 中 | 中 | MAX_HISTORY = 200 上限 |
| 死锁检测漏判 | 中 | 中 | V2 范围覆盖 90% 常见情况，未覆盖时通过 toast 提示 |
| 触屏滑动误触 | 中 | 中 | 阈值 30px |
| Canvas 性能问题 | 低 | 中 | 减少 drawCell 内的循环 / 离屏 canvas 缓存静态层 |
| AudioContext 锁（iOS Safari） | 中 | 低 | 首次用户交互后 resume |
| localStorage 配额超限 | 低 | 低 | 静默吞掉异常 |

## 六、资源投入

### 6.1 人力

- **1 人全栈开发**（前端为主，可独立完成）

### 6.2 时间

- **总工期**：3-4 天（按每天 6-8 小时有效开发计）
